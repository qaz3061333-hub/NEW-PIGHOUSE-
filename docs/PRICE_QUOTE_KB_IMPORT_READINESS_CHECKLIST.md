## 三、已由使用者確認的規則

### A. 洗澡 / 大美容基本項目

- 洗澡與大美容都已包含基本美容項目。
- 已包含修剪指甲及磨圓、修剪腳緣、修剃腳底毛、肛門毛、肚子毛、生殖器毛、擠肛門腺、視情況修眼周雜毛、雜毛修整、廢毛梳整、耳朵清潔。
- 若客人有特別不要做的基本項目，需主動提出。
- 若沒有做洗澡或大美容，只單做某項服務，才用單項計費。

### B. 犬種價格

- 報價前至少需要品種 / 類型與體重。
- 對客應報價格區間，不報單一死價格。
- 報價區間以基礎價 + 體型 / 毛長 / 激動加價上限計算。
- 手吹犬種依三個級距加價，例如 200 / 250 / 300。
- 包月也依相同級距加價邏輯。

### C. 另計

- 另計不是一律建立人工任務。
- 另計代表不可自動算完整總價。
- 可以先回覆可確定的基礎價格，再提醒另計項目需現場評估。
- 若涉及客訴、價格爭議、退款、受傷、流血、醫療疑慮，仍需走高風險人工流程。

### D. 手修

- 手修不是一般洗澡 / 大美容預設項目。
- 只有客人提到全身手修、部分手修、腳柱、屁屁、領巾、手剪造型等，才加入手修邏輯。
- 手修有分全身手修與部分手修，但都落在手修加價區間。
- 若手修標示另計，先報大美容可確定區間，再提醒手修現場另計。

### E. 體外蟲 / 廢毛 / 打結

- 發現活體成蟲，一律 +150 改為除蚤洗劑。
- 超過 10 隻成蟲，收環境清潔費 300～800。
- 蟲量過多可退貨，並收取已完成項目費用。
- 廢毛與打結不提供固定加價範圍，只提醒現場評估另計。

### F. 加購 / 單項

- RENADOG 樂納朵全護潤目前統一 100。
- 剪指甲 100～200，主要依配合度、激動、攻擊風險。
- 磨指甲 50～100，主要依配合度、激動、攻擊風險。
- 清耳朵與拔耳毛可單做，但洗澡 / 大美容已包含。
- 嚴重耳況一律建議就醫。
- 草本泥 / 火山泥只自動報到 20kg 以下，20kg 以上請致電或轉人工。
- 睡墊租用 / 購買只有中和店適用。
- 外出散步可加購，但需自備不易掙脫胸背牽繩，過強攻擊性不可加購。

### G. 貓咪

- 貓咪不是依犬種，而是依短毛 / 中長毛、長毛、巨型貓種分類。
- 巨型貓種，例如緬因貓，一律另計，不自動報價。

## 四、目前仍不可直接匯入正式 KB 的原因

目前仍不可直接把既有文件匯入正式 KB，原因如下：

- 目前仍是草案與 review 文件。
- 尚未決定正式 KB 文章切法。
- 尚未決定一項一篇，還是同類合併成一篇。
- 尚未測試 Conversation Logs / KB query 是否能正確命中。
- 尚未測試缺品種、缺體重、缺膚質、缺分店時是否正確追問。
- 尚未測試另計、手修、體外蟲、醫療疑慮是否正確處理。
- 尚未測試價格爭議、退款、客訴、受傷、流血是否正確轉人工。
- 尚未建立正式 KB 匯入流程。
- 尚未決定是否需要固定欄位，例如 `can_auto_quote`、`needs_onsite_assessment`、`medical_risk`、`branch_limit`。

換句話說，目前這些文件適合當作正式 KB 的候選來源與人工 review 依據，但還不適合直接當成正式資料匯入。

## 五、正式 KB 建立前檢查清單

### A. 資料完整性

- [ ] 每篇 KB 是否有清楚標題
- [ ] 是否有類型，例如犬種價格、貓咪價格、加購項目、住宿外加
- [ ] 是否有適用對象
- [ ] 是否有價格或價格區間
- [ ] 是否有缺資料時的追問方式
- [ ] 是否有現場評估條件
- [ ] 是否有不可自動報價條件
- [ ] 是否有分店限制
- [ ] 是否有醫療 / 客訴 / 退款 / 受傷轉人工提醒

### B. 報價安全性

- [ ] 是否避免系統猜價
- [ ] 是否避免 20kg 以上草本泥 / 火山泥自動外推價格
- [ ] 是否避免巨型貓種自動報價
- [ ] 是否避免手修另計項目自動算完整總價
- [ ] 是否避免嚴重耳況被當成美容服務處理
- [ ] 是否避免體外蟲過量時自動保證可服務
- [ ] 是否避免客訴 / 退款 / 受傷 / 流血自動完整回答

### C. 文章切法

正式 KB 建立前，仍需決定以下文章切法：

- 犬種價格：一犬種一篇，還是多犬種合併？
- 貓咪價格：是否獨立一篇？
- 加購項目：一類一篇，還是每項一篇？
- 住宿外加：是否獨立一篇？
- 體外蟲 / 環境清潔：是否獨立一篇？
- 報價共通提醒：是否獨立一篇？

### D. 測試前置條件

- [ ] 已在 sandbox KB 建立少量測試資料
- [ ] 已確認只查 active KB
- [ ] 已確認 inactive KB 不會被用來回答
- [ ] 已確認 KB 無資料時會建立 Knowledge Gap 或轉人工
- [ ] 已確認不會把整包 KB 丟給 Gemini
- [ ] 已確認只取少量相關 KB 產生草稿
